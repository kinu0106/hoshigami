# ビルドエラーまとめと開発時の注意点

## 今回発生したエラー

### 1. TypeScript型エラー：存在しないプロパティへのアクセス

**エラー内容：**
```
Type error: Property 'department' does not exist on type '{ companyName: string; }'.
```

**発生箇所：**
- ファイル：`src/app/list/page.tsx` の818行目
- コード：`{invoiceRecipientBase.department && <p>{invoiceRecipientBase.department}</p>}`

**原因：**
- `invoiceConfig.ts`で`invoiceRecipientBase`の`department`プロパティが削除されていた（コメントアウト）
- しかし、`page.tsx`ではまだ`department`にアクセスしようとしていた
- 型定義と実際の使用箇所が不一致だった

**修正方法：**
- 不要になった`department`へのアクセスを削除

**開発時の注意点：**
- ✅ **型定義を変更したら、使用箇所も確認する**
  - プロパティを削除・変更した場合、そのプロパティを使用している箇所をすべて確認
  - TypeScriptの型チェックが有効なら、ビルド時にエラーとして検出される
- ✅ **コメントアウトしたプロパティは、使用箇所も削除する**
  - コメントに「削除」と書いてあっても、コードから削除されていない場合はエラーになる
- ✅ **ビルド前に型チェックを実行する**
  - 開発環境で`npm run build`を実行して、型エラーを早期に発見する

### 2. Next.js 15のSuspenseバウンダリエラー

**エラー内容：**
```
useSearchParams() should be wrapped in a suspense boundary at page "/login".
Read more: https://nextjs.org/docs/messages/missing-suspense-with-csr-bailout
```

**発生箇所：**
- ファイル：`src/app/login/page.tsx`
- 原因：`useSearchParams()`フックを使用していたが、Suspenseでラップされていなかった

**原因：**
- Next.js 15では、`useSearchParams()`を使用するコンポーネントはSuspenseバウンダリでラップする必要がある
- これは、サーバーサイドレンダリング（SSR）とクライアントサイドレンダリング（CSR）の整合性を保つため

**修正方法：**
- `useSearchParams()`を使用している部分を別コンポーネントに分離
- そのコンポーネントを`<Suspense>`でラップ

**修正前：**
```typescript
export default function LoginPage() {
  const searchParams = useSearchParams(); // エラー
  // ...
}
```

**修正後：**
```typescript
function LoginForm() {
  const searchParams = useSearchParams(); // OK
  // ...
}

export default function LoginPage() {
  return (
    <Suspense fallback={<div>読み込み中...</div>}>
      <LoginForm />
    </Suspense>
  );
}
```

**開発時の注意点：**
- ✅ **Next.js 15の新機能・変更点を確認する**
  - `useSearchParams()`、`useRouter()`などのフックは、Suspenseでラップが必要な場合がある
  - Next.jsのバージョンアップ時は、変更点を確認する
- ✅ **Next.jsの公式ドキュメントを参照する**
  - エラーメッセージにURLが含まれている場合は、必ず確認する
  - `https://nextjs.org/docs/messages/missing-suspense-with-csr-bailout`
- ✅ **開発環境でビルドを実行して確認する**
  - 本番環境にデプロイする前に、必ず`npm run build`を実行する
  - 開発サーバー（`npm run dev`）では検出されないエラーもある

## 開発時の一般的な注意点

### 1. 型安全性の確保

- **TypeScriptの型チェックを有効にする**
  - `tsconfig.json`で厳格な型チェックを有効にする
  - `"strict": true`などの設定を確認する

- **型定義と実装の整合性を保つ**
  - 型定義を変更したら、使用箇所も確認する
  - 不要になったプロパティは、使用箇所からも削除する

### 2. Next.jsのベストプラクティス

- **Next.jsのバージョンに注意する**
  - バージョンアップ時は、変更点を確認する
  - 特にメジャーバージョンアップ時は、破壊的変更がある可能性が高い

- **SSRとCSRの違いを理解する**
  - サーバーサイドで実行されるコードとクライアントサイドで実行されるコードを区別する
  - `"use client"`ディレクティブの使用を適切に行う

### 3. ビルド前の確認

- **開発環境でビルドを実行する**
  - 本番環境にデプロイする前に、必ず`npm run build`を実行する
  - エラーがあれば、修正してからデプロイする

- **型チェックとリントを実行する**
  - `npm run build`で型チェックとリントが実行される
  - エラーがあれば、修正してからデプロイする

### 4. コードレビューのポイント

- **型定義の変更を確認する**
  - 型定義を変更した場合は、使用箇所も確認する
  - 不要になったプロパティは、使用箇所からも削除する

- **Next.jsの新機能・変更点を確認する**
  - フックの使用方法が変更されていないか確認する
  - Suspenseバウンダリが必要な場合は、適切にラップする

## 今後の開発フロー推奨事項

### 1. 開発時のチェックリスト

- [ ] コードを変更したら、型チェックを実行する
- [ ] ビルドを実行して、エラーがないか確認する
- [ ] 型定義を変更したら、使用箇所も確認する
- [ ] Next.jsのバージョンアップ時は、変更点を確認する

### 2. デプロイ前のチェックリスト

- [ ] `npm run build`を実行して、エラーがないか確認する
- [ ] 型エラーがないか確認する
- [ ] リントエラーがないか確認する
- [ ] 本番環境の環境変数が正しく設定されているか確認する

### 3. エラーが発生した場合の対処

1. **エラーメッセージを確認する**
   - エラーメッセージに含まれるファイル名と行番号を確認する
   - エラーメッセージにURLが含まれている場合は、確認する

2. **該当箇所を確認する**
   - エラーが発生しているファイルと行を確認する
   - 関連する型定義や設定ファイルも確認する

3. **修正方法を検討する**
   - エラーの原因を特定する
   - 適切な修正方法を検討する
   - 修正後、再度ビルドを実行して確認する
