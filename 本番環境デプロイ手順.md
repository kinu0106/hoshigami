# 本番環境デプロイ手順 - Xserver VPS (Ubuntu 24.04)

## 概要

このドキュメントは、Xserver VPS（Ubuntu 24.04）にHOSHIGAMIシステムをデプロイする手順を説明します。

## 複数のシステムを同じVPSに配置する場合

**はい、同じVPSサーバーに複数のシステムを配置することができます。**

例えば、HOSHIGAMIシステムと別のシステム（例：sanchokuシステム）を同じVPSで運用することが可能です。

### 複数システムを配置する場合、ドメインは必要？

**複数システムを配置する場合は、ドメインの取得を強く推奨します。**

**理由：**
- サブドメインで各システムを分けられる（管理が簡単）
- 各システムでSSL証明書を取得可能（セキュリティ向上）
- パスで分ける方法より設定が簡単

詳細は「ドメインとサーバー名について」セクション（後述）を参照してください。

### 複数システムを配置する場合の考慮点

1. **ポート番号の管理**
   - 各システムのバックエンドは異なるポート番号を使用する必要があります
   - 例：HOSHIGAMI（ポート5001）、sanchoku（ポート5002）
   - フロントエンドも異なるポート番号を使用
   - 例：HOSHIGAMI（ポート3000）、sanchoku（ポート3001）

2. **Nginxの設定**
   - ドメインを使用する場合：サブドメインで分ける（推奨、設定例は後述）
   - サーバー名のみの場合：パスで分ける（設定例は後述）

3. **データベースの管理**
   - 各システムで別々のデータベースを作成する（推奨）
   - 例：`hoshigami`データベース、`sanchoku`データベース

4. **リソースの管理**
   - メモリやCPUの使用量を監視
   - 必要に応じてVPSのプランをアップグレード

### 複数システムを配置する場合の設定例

#### 方法1: サブドメインで分ける（推奨）

**Nginx設定例：**

```nginx
# HOSHIGAMIシステム
server {
    listen 80;
    server_name hoshigami.example.com;
    
    location / {
        proxy_pass http://localhost:3000;
        # ... その他の設定
    }
    
    location /api {
        proxy_pass http://localhost:5001;
        # ... その他の設定
    }
}

# sanchokuシステム
server {
    listen 80;
    server_name sanchoku.example.com;
    
    location / {
        proxy_pass http://localhost:3001;
        # ... その他の設定
    }
    
    location /api {
        proxy_pass http://localhost:5002;
        # ... その他の設定
    }
}
```

#### 方法2: パスで分ける

**Nginx設定例：**

```nginx
server {
    listen 80;
    server_name vps-xxx.xserver.jp;
    
    # HOSHIGAMIシステム
    location /hoshigami {
        proxy_pass http://localhost:3000;
        rewrite ^/hoshigami/(.*)$ /$1 break;
    }
    
    location /hoshigami/api {
        proxy_pass http://localhost:5001;
        rewrite ^/hoshigami/api/(.*)$ /api/$1 break;
    }
    
    # sanchokuシステム
    location /sanchoku {
        proxy_pass http://localhost:3001;
        rewrite ^/sanchoku/(.*)$ /$1 break;
    }
    
    location /sanchoku/api {
        proxy_pass http://localhost:5002;
        rewrite ^/sanchoku/api/(.*)$ /api/$1 break;
    }
}
```

**注意**: パスで分ける場合、フロントエンドの設定（Next.jsの`basePath`など）も調整が必要になる場合があります。

#### 方法3: IPアドレスとドメインを混在させる

**同じVPS上で、一部のシステムはIPアドレス、他のシステムはドメインで運用することも可能です。**

**Nginx設定例：**

```nginx
# HOSHIGAMIシステム（IPアドレスでアクセス）
server {
    listen 80;
    server_name 123.456.789.012;  # VPSのIPアドレス
    
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    
    location /api {
        proxy_pass http://localhost:5001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# 他社のシステム（ドメインでアクセス、SSL証明書あり）
server {
    listen 80;
    server_name other-company.example.com;
    
    # HTTPからHTTPSにリダイレクト
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name other-company.example.com;
    
    ssl_certificate /etc/letsencrypt/live/other-company.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/other-company.example.com/privkey.pem;
    
    location / {
        proxy_pass http://localhost:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    
    location /api {
        proxy_pass http://localhost:5002;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**この方法の特徴：**

- **HOSHIGAMIシステム**: `http://123.456.789.012` でアクセス（HTTPのみ、SSL証明書なし）
- **他社のシステム**: `https://other-company.example.com` でアクセス（HTTPS、SSL証明書あり）

**環境変数の設定例：**

```bash
# HOSHIGAMIシステム（IPアドレス使用）
# /var/www/hoshigami/backend/.env.production
CORS_ORIGIN=http://123.456.789.012
NEXT_PUBLIC_API_URL=http://123.456.789.012/api

# 他社のシステム（ドメイン使用）
# /var/www/other-company/backend/.env.production
CORS_ORIGIN=https://other-company.example.com
NEXT_PUBLIC_API_URL=https://other-company.example.com/api
```

**メリット：**
- 各システムの要件に応じて、IPアドレスとドメインを選択できる
- 社内のみのシステムはIPアドレス、外部公開するシステムはドメインと使い分け可能

**デメリット：**
- 設定が複雑になる
- 管理が分散する

### PM2での管理

複数のシステムをPM2で管理する場合：

```bash
# HOSHIGAMIシステム
cd /var/www/hoshigami/backend
pm2 start dist/index.js --name hoshigami-backend
cd /var/www/hoshigami/frontend
pm2 start npm --name hoshigami-frontend -- start

# sanchokuシステム
cd /var/www/sanchoku/backend
pm2 start dist/index.js --name sanchoku-backend
cd /var/www/sanchoku/frontend
pm2 start npm --name sanchoku-frontend -- start

# すべてのプロセスを確認
pm2 list
```

```
/var/www/
├── hoshigami/
│   ├── backend/
│   └── frontend/
└── sanchoku/
    ├── backend/
    └── frontend/
```

### 環境変数の設定

各システムで独立した環境変数ファイルを使用：

```
/var/www/hoshigami/backend/.env
/var/www/sanchoku/backend/.env
```

### バックアップの設定

複数システムがある場合、バックアップスクリプトも各システムに対応させる：

```bash
#!/bin/bash
BACKUP_DIR="/var/backups"
DATE=$(date +%Y%m%d_%H%M%S)

# HOSHIGAMIデータベースのバックアップ
pg_dump -U hoshigami_user hoshigami > "$BACKUP_DIR/hoshigami_db_$DATE.sql"

# sanchokuデータベースのバックアップ
pg_dump -U sanchoku_user sanchoku > "$BACKUP_DIR/sanchoku_db_$DATE.sql"

# 圧縮
gzip "$BACKUP_DIR/hoshigami_db_$DATE.sql"
gzip "$BACKUP_DIR/sanchoku_db_$DATE.sql"
```

## 前提条件

- Xserver VPS契約済み
- ドメイン設定済み（オプション、SSL証明書取得に必要）
- SSH接続可能
- ローカル環境で動作確認済み

## 設定の順序について

### ドメインを先に決める場合（推奨）

**メリット：**
- 最初から正しい環境変数を設定できる
- SSL証明書の設定を最初から含められる
- 設定のやり直しが少ない

**手順：**
1. ドメインを取得・設定
2. DNS設定でVPSのIPアドレスに紐付け
3. VPSの基本設定（Node.js、PostgreSQLなど）
4. プロジェクトの配置と環境変数設定（ドメインを使用）
5. ビルドと起動
6. Nginx設定（ドメインを使用）
7. SSL証明書の取得

### 先にVPSを設定する場合

**メリット：**
- すぐに動作確認できる（サーバー名やIPアドレスで）
- ドメイン取得に時間がかかる場合でも進められる

**デメリット：**
- 後でドメインを設定する際に環境変数の変更が必要
- Nginx設定の変更が必要
- 設定のやり直しが発生する可能性

**手順：**
1. VPSの基本設定（Node.js、PostgreSQLなど）
2. プロジェクトの配置と環境変数設定（サーバー名またはIPアドレスを使用）
3. ビルドと起動
4. Nginx設定（サーバー名またはIPアドレスを使用）
5. 動作確認
6. 後でドメインを取得・設定
7. 環境変数の変更
8. Nginx設定の変更
9. SSL証明書の取得

### 推奨される順序

**複数システムを配置する予定がある場合：**
→ **ドメインを先に決めることを強く推奨**

**単一システムで、すぐに動作確認したい場合：**
→ 先にVPSを設定してサーバー名で動作確認 → 後でドメインを設定

**本番環境として運用する場合：**
→ **ドメインを先に決めることを推奨**（SSL証明書が必要なため）

**ドメインの決定に時間がかかる場合（推奨される実用的な手順）：**
→ **先にファイルを整理してVPSの基本設定まで進める → ドメイン決定後に環境変数とNginx設定を更新**

この方法のメリット：
- ドメイン決定を待たずに準備を進められる
- ファイル整理や基本設定はドメイン不要
- ドメインが決まったら環境変数とNginx設定だけ変更すればOK

### ドメイン設定のタイミング

ドメインを先に決める場合でも、以下の点に注意：

1. **DNS設定の反映に時間がかかる場合がある**
   - 通常、数分～数時間かかることがある
   - その間はサーバー名やIPアドレスで動作確認可能

2. **ドメインがなくても設定は進められる**
   - 環境変数でサーバー名を使用
   - 後でドメインに変更可能

3. **SSL証明書の取得は最後**
   - ドメインのDNS設定が反映されてから取得
   - それまではHTTPで動作確認

## ドメインとサーバー名について

### サーバー名（ホスト名）とは

**サーバー名**は、Xserver VPSが提供するサーバー自体の名前です。通常、以下のような形式です：
- `vps-xxx.xserver.jp`（Xserverが自動的に割り当てる）
- または、サーバーのIPアドレス（例：`123.456.789.012`）

**サーバー名を使うメリット：**
- ドメインを取得する必要がない（無料）
- すぐに使える（DNS設定が不要）
- テスト環境や社内利用に適している

**サーバー名を使うデメリット：**
- 覚えにくい（`vps-xxx.xserver.jp`のような形式）
- SSL証明書が取得できない（Let's Encryptはドメインが必要）
- HTTPのみの運用になる（セキュリティ上推奨されない）
- プロフェッショナルな印象ではない

**サーバー名の確認方法：**

Xserverのコントロールパネルで確認できます。または、サーバーにSSH接続して以下で確認：

```bash
# ホスト名を確認
hostname

# 完全修飾ドメイン名（FQDN）を確認
hostname -f
```

### ドメインとは

**ドメイン**は、独自に取得したインターネット上の名前です。例：
- `hoshigami.example.com`
- `example.com`
- `hoshigami.jp`

### どちらを使うべきか

#### サーバー名のみを使用する場合

**適しているケース：**
- 開発・テスト環境
- 社内のみの利用（外部公開しない）
- 一時的な利用
- ドメインを取得したくない場合

**アクセス方法：**
- `http://vps-xxx.xserver.jp`（Xserverが割り当てたサーバー名）
- `http://123.456.789.012`（IPアドレス直接）

**制限事項：**
- SSL証明書が取得できない（Let's Encryptはドメインが必要）
- HTTPのみの運用（セキュリティ上推奨されない）
- 覚えにくい

**IPアドレス直打ちでの運用のデメリット：**

1. **SSL証明書の問題**
   - Let's Encryptなどの無料SSL証明書は**ドメイン名が必要**で、IPアドレスには発行できない
   - 自己署名証明書を使うと、ブラウザで**警告が毎回表示**される
   - HTTPSを使う場合、ユーザーが警告を無視する必要がある（セキュリティ意識の低下につながる）
   
   **重要：警告が出る理由について**
   - 警告が出るかどうかは、**VPSか社内サーバーかではなく、証明書の種類**で決まります
   - **自己署名証明書**を使う → 警告が出る（VPSでも社内サーバーでも同じ）
   - **認証局発行の証明書**（Let's Encryptなど）を使う → 警告が出ない
   - **証明書を使わない（HTTPのみ）** → SSL証明書の警告は出ない（ただし、通信が暗号化されていない）
   - IPアドレスで運用する場合、認証局発行の証明書が取得できないため、結果的に自己署名証明書しか使えず、警告が出ることになります
   - ドメインを使えば、Let's Encryptなどの認証局発行の証明書を取得できるため、警告は出ません
   
   **社内サーバーとVPSの違い：**
   - **社内サーバー（オンプレミス）**: 通常、社内ネットワーク内に設置され、**外部から直接アクセスできない**（VPN経由などでアクセス可能にする場合もある）
   - **VPS**: インターネット上に公開されているサーバーで、**外部から直接アクセス可能**
   - 警告の有無は、サーバーの種類（社内/VPS）ではなく、**証明書の種類**で決まります
   - 社内サーバーでも自己署名証明書を使えば警告が出ますし、認証局発行の証明書を使えば警告は出ません
   - **証明書を使わない場合（HTTPのみ）**は、SSL証明書の警告は出ませんが、通信が暗号化されていないという別の問題があります
   
   **IPアドレス + HTTPのみ（証明書なし）で運用する場合：**
   - **SSL証明書の警告は出ません**（証明書を使っていないため）
   - ただし、現代のブラウザ（Chrome、Firefoxなど）では、HTTP接続に対して「安全でない接続」などの警告を表示する場合があります
   - これはSSL証明書の警告とは別物で、通信が暗号化されていないことを示す警告です
   - 社内のみの利用で10人程度であれば、この警告を無視して使用することも可能です
   - **結論：IPアドレスでHTTPのみで運用すれば、SSL証明書の警告は出ません**

2. **IPアドレスの変更時の対応**
   - VPSのIPアドレスが変わった場合、**全員に通知が必要**
   - ブックマークや設定ファイルの更新が必要
   - ドメインを使えば、DNS設定を変更するだけで対応可能

3. **運用面の不便さ**
   - 覚えにくい（例：`http://123.45.67.89`）
   - URLが分かりにくく、ブックマークしても管理しにくい
   - 複数システムを追加する場合、パスで分ける必要がある（例：`http://123.45.67.89/hoshigami`、`http://123.45.67.89/sanchoku`）

4. **将来の拡張性**
   - 複数システムを追加する場合、サブドメインで分ける方が管理しやすい
   - ドメインがあれば、`https://hoshigami.example.com`、`https://sanchoku.example.com`のように分けられる

5. **セキュリティ面**
   - IPアドレスが直接公開される（ただし、限られた人間しか使わないなら大きな問題ではない）
   - HTTP通信は暗号化されないため、パスワードなどの機密情報が平文で送信されるリスクがある

**注意：** 限られた人間しか使わない場合でも、**HTTP通信は暗号化されない**ため、ログイン情報などが盗聴されるリスクがあります。可能であれば、自己署名証明書でもHTTPS化することを推奨します（ブラウザの警告は出ますが、通信は暗号化されます）。

#### ドメインを使用する場合（推奨）

**適しているケース：**
- 本番環境
- 外部公開
- SSL証明書が必要な場合
- 長期的な運用

**アクセス方法：**
- `https://hoshigami.example.com`（SSL証明書あり）
- `http://hoshigami.example.com`（SSL証明書なしの場合）

**メリット：**
- SSL証明書を取得可能（Let's Encryptが無料で利用可能）
- HTTPSで安全に通信できる
- 覚えやすく、プロフェッショナル
- ブランドイメージの向上

### サーバー名を使う場合の設定の違い

サーバー名を使う場合でも、基本的な設定は同じです。違いは以下の点だけです：

1. **環境変数**: `CORS_ORIGIN`と`NEXT_PUBLIC_API_URL`にサーバー名またはIPアドレスを指定（手順6を参照）
2. **Nginx設定**: `server_name`にサーバー名またはIPアドレスを指定（手順11を参照）
3. **SSL証明書**: 取得しない（手順13をスキップ、HTTPのみで運用）

詳細な設定例は、各手順（6. 環境変数の設定、11. Nginxの設定）を参照してください。

### ドメインの設定方法

1. **ドメインを取得**（Xserverで取得するか、他のレジストラで取得）
2. **DNS設定**で、ドメインをVPSのIPアドレスに紐付け
   - Aレコード：`@` または `www` → VPSのIPアドレス
3. **Nginxの設定**で、`server_name`にドメインを指定

**注意**: ドメインがなくても、サーバー名やIPアドレスでアクセス可能ですが、SSL証明書の取得ができません（HTTPのみ）。

### 外部からのアクセスについて

**はい、VPSは外部からアクセス可能です。**

Xserver VPSはインターネット上に公開されているサーバーなので、以下の方法で外部からアクセスできます：

- **ドメインを使用する場合**: `https://your-domain.com` または `http://your-domain.com`
- **サーバー名を使用する場合**: `http://vps-xxx.xserver.jp`
- **IPアドレスを使用する場合**: `http://123.456.789.012`

**アクセス可能にするための要件：**

1. **ファイアウォールの設定**（手順12で設定）
   - ポート80（HTTP）と443（HTTPS）が開放されている必要があります
   - 手順12で`sudo ufw allow 80/tcp`と`sudo ufw allow 443/tcp`を実行します

2. **Nginxの設定**（手順11で設定）
   - Nginxが正しく設定され、ポート80でリッスンしている必要があります

3. **PM2でアプリケーションが起動している**（手順10で設定）
   - バックエンド（ポート5001）とフロントエンド（ポート3000）が起動している必要があります

**社内のみの利用にしたい場合：**

社内のみの利用にしたい場合は、以下の方法があります：

1. **ファイアウォールでIPアドレス制限**
   ```bash
   # 特定のIPアドレスのみ許可（例）
   sudo ufw allow from 192.168.1.0/24 to any port 80
   sudo ufw allow from 192.168.1.0/24 to any port 443
   ```

2. **NginxでIPアドレス制限**
   ```nginx
   # Nginx設定ファイルに追加
   location / {
       allow 192.168.1.0/24;  # 社内ネットワーク
       deny all;
       proxy_pass http://localhost:3000;
   }
   ```

3. **VPN経由でのアクセスのみ許可**

**デフォルトの状態：**
- 手順通りに設定すると、**インターネット上の誰でもアクセス可能**になります
- セキュリティのため、本番環境では必ずログイン機能を使用してください

## 1. 本番環境用ファイル整理

**重要**: この手順はドメインが決まる前でも実行できます。ドメインが決まったら、後で環境変数とNginx設定を更新するだけです。

### 1.1 本番環境に必要なファイル

以下のファイル・フォルダを本番環境に配置します：

```
hoshigami/
├── backend/
│   ├── src/                    # ソースコード（必須）
│   ├── package.json            # 必須
│   ├── package-lock.json       # 必須
│   └── tsconfig.json           # 必須
├── frontend/
│   ├── src/                    # ソースコード（必須）
│   ├── public/                 # 静的ファイル（必須）
│   ├── package.json            # 必須
│   ├── package-lock.json       # 必須
│   ├── tsconfig.json           # 必須
│   └── next.config.ts          # 必須
├── package.json                # ルート設定（必須）
├── package-lock.json           # ルート設定（必須）
└── README.md                   # ドキュメント（推奨）
```

### 1.2 本番環境に不要なファイル

以下のファイル・フォルダは本番環境に配置する必要はありません：

- `node_modules/` - 各ディレクトリ（本番で再インストール）
- `dist/` - バックエンドのビルド成果物（本番で再ビルド）
- `.next/` - フロントエンドのビルド成果物（本番で再ビルド）
- `hoshigami_bk*/` - バックアップフォルダ（別途保管）
- `sanchoku/` - 別プロジェクト（本番環境に配置する場合は残す、しない場合は削除）
- `bk/` - バックアップフォルダ（不要なら削除）
- `.env`, `.env.local` - 環境変数ファイル（本番用は別途作成）

### 1.3 ファイル整理の手順

**ドメインが決まる前でも実行可能です。**以下の手順でファイルを整理してください。

1. **バックアップフォルダを別の場所に移動**
   ```bash
   # ローカルで実行（例）
   # バックアップフォルダを安全な場所に移動
   mv hoshigami_bk20251114 /path/to/backup/
   mv hoshigami_bk20251117 /path/to/backup/
   ```

2. **不要なフォルダの確認**
   - `sanchoku/` - 本番環境に配置する場合は残す、しない場合は削除
   - `bk/` - 不要なら削除

3. **本番環境用のクリーンなディレクトリを作成（推奨）**
   ```bash
   # 本番環境用のディレクトリを作成
   mkdir hoshigami_production
   
   # 必要なファイルのみコピー
   cp -r backend hoshigami_production/
   cp -r frontend hoshigami_production/
   cp package.json hoshigami_production/
   cp package-lock.json hoshigami_production/
   cp README.md hoshigami_production/
   cp プロジェクト計画.md hoshigami_production/
   cp 本番環境デプロイ手順.md hoshigami_production/
   ```

4. **`.gitignore`の確認**
   - 本番環境に不要なファイルが含まれていないか確認
   - `node_modules/`, `dist/`, `.next/`などが除外されているか確認

5. **整理後の確認**
   - 必要なファイルがすべて含まれているか確認
   - 不要なファイルが含まれていないか確認

**注意**: この時点では環境変数ファイル（`.env`）は作成しません。ドメインが決まってから作成します。

## 2. サーバー初期設定

### 2.1 SSH接続

**重要：OSインストール時のSSH keyについて**

OSインストール時にSSH keyの選択オプションがあり、そこで自動でSSH keyが生成される場合があります。その後、新たにSSH keyを追加してしまうと、最初に生成されたSSH keyと異なるため、接続できないトラブルが発生する可能性があります。

**推奨手順：**

1. **OSインストール時に生成されたSSH keyを使用**
   - OSインストール時にSSH keyの選択オプションがある場合は、そこで生成されたSSH keyを使用してください
   - Xserverのコントロールパネルで、OSインストール時に生成されたSSH keyを確認・ダウンロードしてください

2. **SSH接続**

```bash
# SSH keyを使用する場合
ssh -i /path/to/your/private/key root@your-vps-ip
# または
ssh -i /path/to/your/private/key username@your-vps-ip

# パスワード認証の場合
ssh root@your-vps-ip
# または
ssh username@your-vps-ip
```

**注意：** 新たにSSH keyを追加する場合は、OSインストール時に生成されたSSH keyと混同しないよう注意してください。

## 6. 環境変数の設定

### 6.1 バックエンドの環境変数

```bash
cd /var/www/hoshigami/backend
nano .env
```

以下の内容を設定：

```env
PORT=5001
DATABASE_URL=postgresql://hoshigami_user:your-strong-password@localhost:5432/hoshigami
SESSION_SECRET=本番環境用の強力なランダム文字列（32文字以上推奨）
SESSION_COOKIE_NAME=hoshigami.sid
CORS_ORIGIN=https://your-domain.com
# またはサーバー名のみの場合：
# CORS_ORIGIN=http://vps-xxx.xserver.jp
NODE_ENV=production
```

**重要**: `SESSION_SECRET`は強力なランダム文字列を生成してください。

```bash
# ランダム文字列を生成（例）
openssl rand -base64 32
```

### 6.2 フロントエンドの環境変数

```bash
cd /var/www/hoshigami/frontend
nano .env.production
```

以下の内容を設定：

**ドメインを使用する場合：**
```env
NEXT_PUBLIC_API_URL=https://your-domain.com/api
```

**サーバー名のみを使用する場合（HTTPのみ）：**
```env
NEXT_PUBLIC_API_URL=http://vps-xxx.xserver.jp/api
# または
NEXT_PUBLIC_API_URL=http://123.456.789.012/api
```

**注意**: 
- フロントエンドの環境変数は`NEXT_PUBLIC_`で始まるもののみクライアント側で使用可能です
- ドメインを使用する場合は`https://`、サーバー名のみの場合は`http://`を使用してください

### 6.3 ドメインが決まった後の環境変数更新手順

**ドメインが決まる前にサーバー名で設定していた場合、以下の手順で更新してください：**

1. **バックエンドの環境変数を更新**
   ```bash
   cd /var/www/hoshigami/backend
   nano .env
   ```
   ```env
   # 変更前（サーバー名）
   # CORS_ORIGIN=http://vps-xxx.xserver.jp
   
   # 変更後（ドメイン）
   CORS_ORIGIN=https://your-domain.com
   ```

2. **フロントエンドの環境変数を更新**
   ```bash
   cd /var/www/hoshigami/frontend
   nano .env.production
   ```
   ```env
   # 変更前（サーバー名）
   # NEXT_PUBLIC_API_URL=http://vps-xxx.xserver.jp/api
   
   # 変更後（ドメイン）
   NEXT_PUBLIC_API_URL=https://your-domain.com/api
   ```

3. **フロントエンドを再ビルド**
   ```bash
   cd /var/www/hoshigami/frontend
   npm run build
   ```

4. **PM2で再起動**
   ```bash
   pm2 restart hoshigami-backend
   pm2 restart hoshigami-frontend
   ```

5. **Nginx設定を更新**（手順11を参照）
   - `server_name`をドメインに変更
   - 設定をテストして再起動

6. **SSL証明書を取得**（手順13を実行）

## 11. Nginxのインストールと設定（リバースプロキシ）

### 11.1 Nginxのインストール

```bash
sudo apt install -y nginx
```

### 11.2 設定ファイルの作成

```bash
sudo nano /etc/nginx/sites-available/hoshigami
```

以下の設定を記述：

```nginx
server {
    listen 80;
    # ドメインを使用する場合
    server_name your-domain.com;
    # またはサーバー名のみの場合
    # server_name vps-xxx.xserver.jp;
    # またはIPアドレスの場合
    # server_name 123.456.789.012;

    # クライアントの最大ボディサイズ（PDF生成用）
    client_max_body_size 10M;

    # フロントエンド
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }

    # バックエンドAPI
    location /api {
        proxy_pass http://localhost:5001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

### 11.3 設定ファイルの有効化

```bash
# シンボリックリンクを作成

# デフォルトの設定を無効化（必要に応じて）

```

## 13. SSL証明書の設定（Let's Encrypt）

**重要**: SSL証明書の取得には**ドメインが必要**です。サーバー名のみでは取得できません。

### 13.1 Certbotのインストール

```bash
sudo apt install -y certbot python3-certbot-nginx
```

### 13.2 SSL証明書の取得

**ドメインを使用する場合のみ：**

```bash
sudo certbot --nginx -d your-domain.com
```

証明書取得時に以下を設定：
- メールアドレス（証明書の有効期限通知用）
- 利用規約への同意
- リダイレクト設定（HTTP→HTTPS推奨）

**サーバー名のみの場合：**
- SSL証明書は取得できません
- HTTPのみで運用することになります（セキュリティ上推奨されません）
- 社内のみの利用やテスト環境でのみ使用してください

### 13.3 自動更新の確認

```bash
# テスト実行
sudo certbot renew --dry-run

# 自動更新はsystemd timerで設定済み（確認）
systemctl status certbot.timer
```

## 14. 定期バックアップの設定

### 14.1 バックアップスクリプトの作成

```bash
sudo nano /usr/local/bin/hoshigami-backup.sh
```

以下の内容を記述：

```bash
#!/bin/bash
BACKUP_DIR="/var/backups/hoshigami"
DATE=$(date +%Y%m%d_%H%M%S)

# バックアップディレクトリの作成
mkdir -p $BACKUP_DIR

# データベースのバックアップ
sudo -u postgres pg_dump -U hoshigami_user hoshigami > "$BACKUP_DIR/db_$DATE.sql"

# バックアップファイルを圧縮
gzip "$BACKUP_DIR/db_$DATE.sql"

# 古いバックアップを削除（30日以上）
find $BACKUP_DIR -name "db_*.sql.gz" -mtime +30 -delete

echo "バックアップ完了: db_$DATE.sql.gz"
```

### 14.2 実行権限の付与

```bash
sudo chmod +x /usr/local/bin/hoshigami-backup.sh
```

### 14.3 定期実行の設定（crontab）

```bash
sudo crontab -e
```

以下の行を追加（毎日午前2時に実行）：

```
0 2 * * * /usr/local/bin/hoshigami-backup.sh >> /var/log/hoshigami-backup.log 2>&1
```
